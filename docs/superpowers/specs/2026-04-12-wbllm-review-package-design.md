# Design Spec: WB LLM Review Package — v8 Output Quality Review

**Date:** 2026-04-12
**Status:** Approved
**Purpose:** Generate a review package for the WBG internal LLM to assess the quality and sufficiency of FCV Project Screener v8 outputs against internal WBG guidance

---

## Context

Previous WB LLM review rounds (20260411) focused on:
- Round 1: Knowledge base content accuracy
- Round 2: MPA and IPF-DDO instrument deep-dive
- Round 3: Prompt quality and completeness

This round (20260412) shifts focus to **output quality** — does what the app actually produces meet WBG FCV standards? The LLM will receive both the reference documents (what the app knows) and the actual stage outputs (what it produced) to assess end-to-end quality.

## Generation Script

**File:** `app_feedback/20260412_wbllm_review/generate_review_docs.py`

**Approach:** Import `background_docs` and extract `DEFAULT_PROMPTS` from `app.py` directly (both are Python modules in the repo root). Use `python-docx` to generate formatted `.docx` files.

**Run from:** Repo root — `python app_feedback/20260412_wbllm_review/generate_review_docs.py`

**Script structure:**
```
generate_review_docs.py
├── import background_docs (WB_INSTRUMENT_GUIDE, WB_PROCESS_GUIDE, FCV_GLOSSARY, FCV constants)
├── extract DEFAULT_PROMPTS from app.py (via ast.parse or regex — avoids importing Flask)
├── write_knowledge_base_doc()    → 01_Knowledge_Base_v8.docx
├── write_stage_prompts_doc()     → 02_Stage_Prompts_v8.docx
├── write_fcv_constants_doc()     → 03_FCV_Reference_Constants_v8.docx
└── main()
```

**Docx formatting:**
- Title style for document heading
- Heading 1 for major sections (e.g., instrument name, stage name, constant name)
- Heading 2 for sub-sections (e.g., FCVLevers, Not Applicable, etc.)
- Normal style for body text
- Consistent with 20260411 document structure

**Extracting DEFAULT_PROMPTS safely:** Use `ast.parse()` on `app.py` to extract the `DEFAULT_PROMPTS` dictionary without executing Flask/Anthropic imports. Fall back to regex if AST approach fails.

## Feedback Prompt Design

**File:** `PROMPT_Output_Quality_Review.md`

The prompt accompanies all three reference docs + the three stage output docs. It asks the WB LLM to:

### Section 1 — Stage 1 Extraction Quality
- Does the extraction correctly identify the instrument type, document type, approval/preparation date, safeguards framework?
- Is the project context (country, sector, PDO) accurately captured?
- Are the temporal context markers correct and appropriately applied?
- Does the citation tiering (Tier 1 uploaded docs / Tier 2 web research / Tier 3 training knowledge) appear consistent?

### Section 2 — Stage 2 Assessment Quality
- Is the FCV assessment grounded in the FCV Refresh framework (4 shifts)?
- Are the S/R tags (Sensitivity/Responsiveness) correctly applied?
- Does the assessment apply instrument-appropriate criteria (i.e., not penalising an IPF for things only possible under a DPO)?
- Is the Do No Harm traffic-light accurate based on the project content?
- Are the ratings (Sensitivity/Responsiveness) appropriately calibrated for this instrument and context?
- Are there any contradictions or logical inconsistencies?
- Do the 6 supplementary dimensions (Gender/GBV, Climate-FCV, PEA, HDP nexus, IDA FCV Envelope, digital risks) add value or introduce noise?

### Section 3 — Stage 3 Recommendation Quality
- Are the priority recommendations specific (geography, mechanism, entry points)?
- Are they feasible under the identified instrument (IPF)?
- Are they grounded in the project document rather than generic FCV guidance?
- Is the Horizon Considerations section appropriate — does it correctly separate in-scope from beyond-scope items?
- Are the suggested PAD section labels and RF indicator specifications correct and useful?
- Is the recommended language (suggested_language in actions[]) ready to use or too generic?

### Section 4 — Overall Sufficiency
- Would an FCV practitioner find this output sufficient to inform real design decisions?
- What is missing or misleading?
- What should be strengthened?

**Prompt tone:** Direct, instructional. Provide the reference docs as context ("here is what the app knows") and the stage outputs ("here is what it produced") and ask for an honest practitioner-level critique. Request specific, actionable feedback.
