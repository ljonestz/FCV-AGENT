# FCV Rating Consistency — Structured Rubric Design Spec

**Date:** 2026-04-03
**Problem:** Repeated runs of the same PAD produce wildly different sensitivity/responsiveness ratings (e.g., "Adequate" vs "Very Low") because the prompts contain no scoring rubric — the LLM vibes a rating from its analysis.
**Solution:** Add an explicit, formula-based scoring rubric to the Stage 2 prompt. Stage 3 inherits Stage 2's ratings.

---

## Rating Scale (unchanged)

Extremely Low | Very Low | Low | Adequate | Well Embedded | Very Well Embedded

---

## Sensitivity Rating Formula

Based on how many of the 12 OST recommendations are addressed from a sensitivity perspective. The LLM must count from its own Under the Hood 12-rec table (which it generates before the rating).

| OST Recs Addressed (S-relevant) | Baseline Rating |
|---|---|
| 0–2 | Extremely Low |
| 3–4 | Very Low |
| 5–6 | Low |
| 7–8 | Adequate |
| 9–10 | Well Embedded |
| 11–12 | Very Well Embedded |

"Addressed" means the rec is rated "Addressed" or "Partially Addressed" in the 12-rec table. "Gap" does not count.

### Sensitivity Quality Gates

These conditions cap the sensitivity rating regardless of the count:

| Condition | Cap |
|---|---|
| 3+ Do No Harm principles rated "Gap" | Capped at Low |
| No conflict/security analysis in project design | Capped at Adequate |
| No geographic specificity in targeting | Capped at Adequate |

Gates are checked in order. The most restrictive cap wins. If the baseline from the count is already at or below the cap, no change.

---

## Responsiveness Rating Formula

Based on alignment with the 4 FCV Refresh shifts (Anticipate, Differentiate, Jobs & Private Sector, Enhanced Toolkit) plus evidence of active root-cause engagement measures.

| Shifts Addressed + Active Measures | Baseline Rating |
|---|---|
| 0 shifts, no active measures | Extremely Low |
| 1 shift, minimal measures | Very Low |
| 1–2 shifts, some concrete measures | Low |
| 2–3 shifts, concrete measures with specificity | Adequate |
| 3–4 shifts, strong embedded measures | Well Embedded |
| 4 shifts, deeply embedded throughout design | Very Well Embedded |

"Addressed" means the project design contains concrete, specific measures that align with the shift — not just a passing mention.

### Responsiveness Quality Gates

| Condition | Cap |
|---|---|
| Zero FCV Refresh shift alignment | Capped at Very Low |
| No adaptive M&E for FCV dynamics | Capped at Low |

---

## Rating Reasoning Block

The Stage 2 prompt requires the LLM to emit a hidden reasoning block showing its step-by-step scoring. This block is stripped from display but available for auditing.

Format:
```
%%%RATING_REASONING_START%%%
SENSITIVITY SCORING:
- Recs addressed (S-relevant): [list rec numbers] → count: X/12
- Baseline from count: [rating]
- Quality gate check:
  - DNH gaps: [count] → [cap or no cap]
  - Conflict/security analysis present: [yes/no] → [cap or no cap]
  - Geographic specificity: [yes/no] → [cap or no cap]
- Most restrictive cap: [rating or "none"]
- FINAL SENSITIVITY RATING: [rating]

RESPONSIVENESS SCORING:
- Shifts addressed: [list shifts] → count: X/4
- Active root-cause measures: [brief description]
- Baseline from shifts + measures: [rating]
- Quality gate check:
  - Any shift alignment: [yes/no] → [cap or no cap]
  - Adaptive M&E for FCV: [yes/no] → [cap or no cap]
- Most restrictive cap: [rating or "none"]
- FINAL RESPONSIVENESS RATING: [rating]
%%%RATING_REASONING_END%%%
```

Key instruction to LLM: **"Do not use your general impression. Follow the scoring formula below step by step. Show your work in the rating reasoning block."**

## Stage 2 Prompt Changes

Insert the full rubric (sensitivity formula + gates, responsiveness formula + gates, reasoning block format) into `DEFAULT_PROMPTS["2"]` immediately before the existing `# Ratings Block` section.

Replace the existing rating instructions:
```
Rating scale (use exactly one of): Extremely Low | Very Low | Low | Adequate | Well Embedded | Very Well Embedded
```

With the full rubric tables + quality gates + reasoning block format + the instruction to follow the formula mechanically.

The rubric references the LLM's own Under the Hood output (12-rec table, DNH checklist) which it has already generated by this point in the prompt.

---

## Stage 3 Prompt Changes

Remove all independent rating generation instructions. Replace with:

> "For `fcv_rating` and `fcv_responsiveness_rating`: use the sensitivity and responsiveness ratings from Stage 2 exactly as provided in the conversation history. Copy them into the JSON fields without modification. Do not re-assess or override the Stage 2 ratings."

This eliminates the current problem where Stage 2 and Stage 3 can produce contradictory ratings.

## Backend Changes

### `extract_stage2_ratings()` in `app.py`

Add extraction of the rating reasoning block:

```python
def extract_stage2_ratings(stage2_output):
    # Existing: extract %%%STAGE2_RATINGS_START/END%%% → {sensitivity_rating, responsiveness_rating}
    # New: also extract %%%RATING_REASONING_START/END%%% → rating_reasoning (string)
    # Return: {error, sensitivity_rating, responsiveness_rating, rating_reasoning}
```

The `rating_reasoning` is included in the SSE done event for Stage 2. Not displayed to the user — available for debugging/auditing.

### `/api/run-stage` route (Stage 3)

No code change needed. The Stage 2 output (including ratings) is already in the conversation history passed to Stage 3. The prompt change handles the rest.

### `/api/run-express` route

No code change needed. Stage 2 output flows into Stage 3 context via the same history mechanism.

## Expected Outcome

- Same PAD run 3 times should produce the same or adjacent ratings (e.g., "Low" and "Low", or "Low" and "Adequate" — never "Very Low" and "Well Embedded")
- Rating reasoning block makes scoring auditable
- Stage 2 and Stage 3 ratings always match (single source of truth)

4. Inspect rating reasoning blocks to verify the formula was followed
